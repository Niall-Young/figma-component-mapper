# 页面模式识别

这份文档用于处理“单个节点看不清，但整块区域能看懂”的场景。

## 识别原则

- 老设计稿往往组件粒度不规范，应允许从页面结构反推“布局 + 组件”组合模式
- 页面模式优先用于列表页、筛选页、弹窗表单页、详情页操作区
- 页面模式识别结果可以是“布局 + 组件”的组合，而不必强行映射成单一组件

## 筛选区 + 表格 + 分页

典型信号：

- 顶部一排 Input、Select、DatePicker 候选
- 旁边有“查询/重置/新增”等操作按钮
- 中间是多列表格结构
- 底部有分页

推荐输出：

- 筛选区：表单布局 + 表单项 + 输入类组件
- 数据区：`Table`
- 底部：`Pagination`

## Dialog 表单页

典型信号：

- 页面中央浮层容器
- 顶部有标题
- 中间是表单项列表
- 底部有取消/确定按钮组

推荐输出：

- 外层：`Dialog`
- 内容：表单布局 + 表单控件
- 底部：footer action 区 + 按钮组

## 表格操作列

典型信号：

- 最右侧窄列
- 反复出现“编辑/删除/查看/更多”
- 文案多为短文本或 text action 风格

推荐输出：

- `Table` 的操作列
- 内部动作优先使用组件库已有的轻量操作样式

## 输出要求

如果识别结果是页面模式，应在输出中明确区分：

- 哪些部分是组件
- 哪些部分是布局容器
- 哪些部分可能需要业务层拼装
