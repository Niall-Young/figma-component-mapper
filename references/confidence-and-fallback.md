# 置信度与降级输出规则

为了避免在设计稿不规范时输出伪精确结果，这份 skill 必须显式管理置信度。

## 置信度层级

### High

适用条件：

- 文本、视觉、布局三类证据中至少两类强命中
- 组件类型清晰
- 关键能力映射有足够证据支撑

允许输出：

- 直接给出目标组件
- 直接给出高置信度能力映射
- 说明哪些字段仍为推断

### Medium

适用条件：

- 组件类型大致明确，但部分能力映射不稳定
- 或只有页面上下文支持，单节点证据较弱

允许输出：

- 推荐组件
- 推荐能力候选
- 清楚列出待确认点

不允许：

- 把中等置信度结果说成最终确定实现

### Low

适用条件：

- 证据单薄
- 文本与视觉冲突明显
- 只能看出是一个交互区块，看不出具体组件

允许输出：

- 语义判断
- 候选组件列表
- 不确定原因

不允许：

- 直接产出确定代码
- 虚构组件库 API

## name_dependency

用于说明这次判断有多依赖命名：

- `none`：完全不依赖图层名
- `low`：图层名仅作辅助确认
- `medium`：图层名提供了重要补充信息
- `high`：主要靠图层名才能成立，这类结果应谨慎使用

默认目标：

- 高频组件尽量做到 `none` 或 `low`

## 推荐输出写法

- 明确写出命中的证据来源
- 对低置信度能力单独降级，不要因为某个字段不确定就否定整个组件
- 如果设计稿视觉与组件库默认样式存在偏差，要写入 `fallback_notes` 或 `deviation_notes`
- 即使某个能力在两个候选之间不完全确定，也应先输出候选，而不是直接跳到样式覆盖
- 只有当多个合法能力组合都无法覆盖设计差异时，才允许进入 `style_override_strategy`

## 能力映射与样式覆盖的置信度处理

允许出现这种组合：

- 组件身份 `high`
- 某些能力 `medium`
- `deviation_notes` 存在

这不意味着应该直接改写 CSS。  
默认仍应先输出能力候选，再说明剩余偏差。
